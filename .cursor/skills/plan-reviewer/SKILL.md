## Шаг 1 — Структурный анализ

Проверь наличие обязательных секций:
- Название, Цель, Границы (входит/не входит)
- Ключевые гарантии и инварианты
- Edge-cases
- Mermaid схема
- Человекочитаемый флоу
- Структура и публичный API
- Тесты
- DoD
- Зависимости, Артефакты, Владелец контракта

Для каждой отсутствующей секции — отметить как пробел.

---

## Шаг 2 — Проверка уровня детализации

Применяй правило: всё что отвечает на «что и почему» — уместно в плане.
Всё что отвечает на «как именно реализовать» — избыточно.

Отметить как избыточное:
- Конкретные значения переменных и параметров функций в описании логики
- Сигнатуры функций с типами аргументов в YAML todos
- Конкретные входные данные и ожидаемые значения в описании тестов
- Имена log-событий с полным списком полей (достаточно категорий)
- SQL-тексты и шаблоны (артефакт реализации, не плана)
- Код или псевдокод внутри описания задач

Отметить как недостаточное:
- Инварианты без явной формулировки условия
- Edge-cases без описания ожидаемого поведения
- Публичный API без указания кто потребитель
- DoD без измеримых критериев
- Todo bootstrap/routes без Lifecycle или без указания потребителя wire (main.py)
- Новая интеграция (SDK, Mongo collection) без аналога в репо и без флоу

### 2.1. Пригодность к task-packager (опционально, если план апрувлен или перед упаковкой)

План модуля **не обязан** содержать код. Проверить, хватит ли материала для task-файлов:

**Достаточно для task без «план читать: да»:**
- секция «Структура» перечисляет файлы по todo;
- контракты с полями/инвариантами в «Публичный API»;
- в репо есть явный аналог для типовых задач (store, bootstrap, routes).

**Недостаточно — правка в план модуля (не раздувать tasks кодом):**
- новый внешний SDK / протокол без секции «Lifecycle» или флоу;
- todo ссылается на файлы, которых нет в «Структура»;
- bootstrap/wire main не описан, а todo `*-bootstrap` есть.

**Избыточно в плане (при упаковке перенести в task как якорь, убрать из плана):**
- пошаговая инструкция «создать функцию X» — оставить в task «Решения / якоря»
  + ссылку на аналог в репо.

В правках указывать: «добавить в план» vs «заполнится при task-packager».

## Шаг 1b — Ревью task-пакета (если передан путь к `index.md`)

Проверить:
- папка = basename `module-*.plan.md` (не короткий slug из YAML `name:`);
- у каждой todo ровно один task-файл;
- у нетривиальных задач заполнены «Решения / якоря» или `план читать: да`;
- нет кода/сигнатур в task-файлах (кроме одной граничной factory при необходимости);
- `depends_on` без циклов и битых id.

Не переоценивать модульный план — только целостность пакета.

## Шаг 3 — Проверка согласованности с другими модулями

Для каждого связанного плана проверить:

**Контракты сущностей:**
- Поля и типы совпадают с планом-владельцем
- Импорты идут через модуль-владелец, не дублируются
- Новые поля добавлены в план-владелец, не в потребителя

**Зависимости:**
- Upstream-модули указаны корректно
- Нет ссылок на модули которые ещё не готовы по порядку реализации
- Открытые вопросы которые закрыты в других модулях — помечены закрытыми

**Инварианты проекта (из M0, актуальный список — индексный план из `memory.md` — раздел «Текущий релиз»):**
- MySQL read-only соблюдён
- tick_id через contextvars упомянут если модуль вызывается в контексте тика
- Часовой пояс MSK (не UTC) для всех datetime
- Async-only публичный API
- Пароли и секреты не попадают в логи

**Терминология:**
- Имена сущностей совпадают с M0 §2 — таблица владельцев (путь к M0 — см. `memory.md`, раздел «Текущий релиз»)
- Имена модулей совпадают с M0 §2 — таблица модулей
- Статусы и литералы совпадают с контрактом владельца

## Шаг 4 — Выявление архитектурных пробелов

Проверить наличие ответов на следующие вопросы:

**Lifecycle:**
- Как модуль инициализируется (lifespan / bootstrap / singleton)?
- Как модуль завершает работу (graceful shutdown)?
- Идемпотентен ли init?

**Ошибки:**
- Какие исключения модуль поднимает наружу?
- Какие исключения глотает и логирует?
- Что происходит при недоступности внешних зависимостей?

**Границы:**
- Есть ли риск что другой модуль возьмёт на себя ответственность этого?
- Есть ли неявные зависимости которые не указаны в секции «Зависимости»?

**Конфигурация:**
- Все параметры из env описаны с дефолтами?
- Валидация env при старте — fail-fast?

**Observability:**
- tick_id включается в логи если модуль вызывается в контексте тика?
- Секреты замаскированы в логах?

## Шаг 5 — Числовая оценка

Оценить план по пяти критериям, каждый 0–2 балла, сумма — итоговая оценка из 10:

| Критерий | 0 баллов | 1 балл | 2 балла |
|---|---|---|---|
| Полнота структуры (Шаг 1) | пропущено ≥2 обязательных секции | пропущена 1 | все на месте |
| Уровень детализации (Шаг 2) | избыточность или недостаточность в ≥3 местах | в 1–2 местах | план на уровне «что и почему» без потерь |
| Согласованность (Шаг 3) | реальные несогласованности с владельцами контрактов | терминологические шероховатости без риска | полностью согласован |
| Архитектурные пробелы (Шаг 4) | ≥2 неотвеченных вопроса из критичных (lifecycle/ошибки) | 1 неотвеченный | все ключевые вопросы закрыты |
| Краткость относительно содержательности | план длиннее необходимого ИЛИ упускает суть в попытке быть кратким | небольшой перекос в одну сторону | оптимальный баланс — нет лишнего, нет недостающего |

Итоговая оценка = сумма. Привести оценку явно: `Оценка: X/10`.

## Шаг 6 — Формирование вывода (однократное ревью)

### Блок 1 — Оценка
`Оценка: X/10`. Коротко: что сделано хорошо, что слабо.

### Блок 2 — Список правок
Каждая правка:
> **ПРАВКА N — <название>**
> Файл: <модуль>, секция: <§N>
> Действие: <добавить | заменить | удалить>
> Текст: <точный текст правки>
> Обоснование: <одно предложение>

### Блок 3 — Промпт для применения правок
Готовый промпт для точечного редактирования плана. Правки не переписывают
план целиком, не меняют структуру без явного указания.

---

## Шаг 7 — Цикл до оценки 9/10 (только если выбран режим «цикл»)

1. Выполнить Шаги 1–5, получить оценку.
2. Если оценка ≥ 9/10 — остановиться, показать финальный Блок 1 и список
   оставшихся (необязательных) замечаний, если есть.
3. Если оценка < 9/10:
   - Применить правки из Блока 2 самостоятельно к плану (это пред-апрувная
     самопроверка, не финальный апрув пользователя на сам план — правки
     допустимы без отдельного разрешения, так как план как целое ещё не
     апрувлен).
   - Повторить Шаги 1–5 на обновлённом плане.
4. Лимит — 5 итераций. Если после 5 итераций оценка не достигла 9/10 —
   остановиться, показать текущую оценку, текущий план и объяснить, какие
   именно критерии не удаётся закрыть и почему (например, реальная
   неустранимая неоднозначность, требующая решения пользователя).
5. По завершении цикла (успешного или по лимиту) — показать итоговую
   историю оценок по итерациям (например: `7/10 → 8/10 → 9/10`), чтобы
   пользователь видел прогресс, а не только финал.
6. Финальный апрув плана как целого — всё равно требует явного
   подтверждения пользователя (см. `planning.mdc` — раздел «Апрув — жёсткие ограничения»).
   Достижение 9/10 закрывает только цикл ревью, не заменяет апрув.

## Ограничения

- Не предлагать правки которые меняют архитектурные решения — только фиксируют
  или уточняют уже принятые
- Не добавлять новые секции если они не требуются шаблоном модульного плана
- Не переформулировать то что уже написано корректно
- Правки по согласованности — только если несогласованность реальная,
  не терминологическая
- Если план соответствует уровню «что и почему» без избыточности — сказать явно,
  не искать правки ради правок
- В цикле (Шаг 7) не наращивать объём плана ради закрытия пробелов — если
  пробел можно закрыть одним предложением, не закрывать его абзацем
- Ревью task-файлов — отдельный проход (Шаг 1b, `task-packager`, `task_template.md`);
  plan-reviewer по умолчанию оценивает модульный план, не заменяет ревью каждого task
