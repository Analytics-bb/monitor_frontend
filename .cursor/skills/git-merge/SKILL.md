# Протокол мержа в main

## Шаг 1 — Проверить состояние

```bash
git status
git branch --show-current
```

Убедиться что все изменения в ветке закоммичены. Если есть незакоммиченные
изменения — использовать навык `git-commit` перед продолжением.

## Шаг 2 — Если мержим не из main (chore → main → feat)

Сначала влить main в текущую ветку чтобы разрешить конфликты локально:

```bash
git fetch origin main
git merge origin/main
```

Если есть конфликты — разрешить, закоммитить, затем продолжать.

## Шаг 3 — Запустить CI

```bash
make ci
```

Если `make ci` упал:
- Ошибки линтера (ruff) — исправить автоматически: `make format && make lint`
- Если после автофикса остались ошибки — исправить вручную, закоммитить через навык `git-commit`
- Ошибки тестов — остановиться и сообщить пользователю, не мержить с упавшими тестами

## Шаг 4 — Вывести план мержа

Показать пользователю до выполнения:

```
ветка:  feat/my-feature → main
метод:  squash merge
коммит: feat(область): описание всех изменений ветки
```

Сообщение squash-коммита формулировать по тем же правилам что в навыке `git-commit` —
одна строка, повелительное наклонение, до 72 символов.

## Шаг 5 — Выполнить squash merge

```bash
git checkout main
git pull origin main
git merge --squash <ветка>
git commit -m "тип(область): описание"
```

После коммита вывести `git log --oneline -3`.

## Ограничения

- Не мержить если `make ci` упал с ошибками тестов
- `git push origin main` — только по явной просьбе пользователя
- Не использовать `--force` при мерже
