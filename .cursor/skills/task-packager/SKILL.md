# Протокол сборки task-пакета

## Шаг 0 — Проверить апрув

Убедиться, что план модуля апрувлен (явное подтверждение пользователя в истории
диалога или статус, зафиксированный в плане). Если апрува не было — остановиться,
спросить пользователя (см. `planning.mdc` — раздел «Апрув — жёсткие ограничения»).

## Шаг 1 — Прочитать план модуля целиком

Путь: `.cursor/plans/<release>/<module-id>.plan.md`, где `<release>` — из `memory.md`
(раздел «Текущий релиз», поле «Папка планов»).
Извлечь: список `todos` (YAML-шапка), секцию «Структура», «Публичный API»,
«Зависимости», «Владелец контракта», «DoD», «Тесты».

## Шаг 2 — Построить граф задач

Для каждой todo определить:
- какие файлы она создаёт/меняет (из секции «Структура» плана);
- зависит ли она от других todo этого же модуля (по порядку из плана,
  по упоминаниям "после", "использует модель из", и т.п.);
- зависит ли от задач других модулей (по секции «Зависимости» плана).

Если порядок неочевиден из текста плана — спросить пользователя, не угадывать.

## Шаг 3 — Создать папку и task-файлы

Путь task-пакета: `.cursor/plans/<release>/<module-plan-basename>/`,
где `<release>` — из `memory.md`, а `<module-plan-basename>` — **имя файла
плана модуля без `.plan.md`**.

Примеры:
- план `module-14-agent-executor.plan.md` → папка `module-14-agent-executor/`
- план `module-8-audit-store-r2.plan.md` → папка `module-8-audit-store-r2/`

**Не использовать** короткий slug из поля `name:` в YAML (`agent-executor`) —
только basename файла плана. Папка однозначно соответствует плану в дереве.

Для каждой todo — файл `<task-id>.plan.md` по `task_template.md`:
- `id` = id todo из плана;
- `parent_module` = путь к module-N.plan.md;
- `depends_on` = id задач этого/других модулей, определённые на Шаге 2;
- `план читать` = "да, §N" только если задаче нужен контекст за пределами
  того, что помещается в сам task-файл (сложный edge-case, неочевидный
  инвариант) — иначе "нет";
- секции «Файлы», «Контракт», «Решения / якоря реализации», «Тесты», «DoD» —
  заполнить кратко, без пересказа всего плана, только то что нужно для
  выполнения именно этой задачи;
- секция «Решения / якоря реализации»:
  - минимум 1 bullet, если задача трогает bootstrap, store, внешний SDK, routes в main;
  - иначе явно «N/A»;
  - якоря брать из плана модуля и существующего кода, не выдумывать API.

Не копировать в task-файл текст, не относящийся к задаче. Цель — task-файл
читается за секунды, не пересказывает план.

## Шаг 4 — Создать index.md

Путь: `.cursor/plans/<release>/<module-plan-basename>/index.md`.

Содержимое:
- ссылка на родительский план;
- таблица задач: id, краткое содержание, depends_on, статус (зеркало
  статуса из YAML-шапки родительского плана — не отдельный источник правды);
- mermaid-граф зависимостей задач;
- явная отметка, какие задачи можно вести параллельно (без общих
  depends_on и без общего владения одним файлом/контрактом).

## Шаг 5 — Проверить целостность пакета

- Имя папки совпадает с basename родительского `module-*.plan.md`.
- Каждая todo плана имеет ровно один task-файл.
- Все id в `depends_on` существуют либо в этом пакете, либо как уже
  завершённые задачи других модулей (проверить статус в их планах).
- Нет циклов в графе зависимостей.
- У нетривиальных задач заполнены «Решения / якоря» или `план читать: да`.

Если есть нарушение — исправить самостоятельно, если решение очевидно
(например, опечатка в id), иначе остановиться и спросить пользователя.

## Шаг 6 — Отчитаться

Показать пользователю список созданных файлов и mermaid-граф из index.md.
Не коммитить — создание task-пакета не требует отдельного коммита,
коммиты делаются на уровне задач (task-runner) и закрытия модуля (module-closer).
Спросить: «Граф зависимостей и порядок задач корректны?»
