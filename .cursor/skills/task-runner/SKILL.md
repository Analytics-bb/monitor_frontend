# Протокол реализации задачи

## Шаг 0 — Проверить чистоту рабочей копии

Если есть незакоммиченные изменения не от текущей задачи — остановиться, сообщить пользователю.

## Шаг 1 — Найти task-файл

Путь: `.cursor/plans/<release>/<module-plan-basename>/<task-id>.plan.md`,
где `<release>` — `FE` / `R2` из `memory.md`, а `<module-plan-basename>` —
имя файла плана модуля без `.plan.md` (см. `task-packager`).
Прочитать task-файл полностью.

## Шаг 2 — Определить нужен ли план модуля

Читать полный план `.cursor/plans/FE/<module>.plan.md` (или `R2/`) ТОЛЬКО если task-файл содержит:
- явную пометку `план читать: §N`
- ссылку вида `см. план §...`

Если `план читать: нет` — план не открывать.

## Шаг 3 — Проверить зависимости

Если task-файл содержит `зависимости: m7-xxx, m7-yyy` — для каждой проверить
статус `completed` в YAML-шапке `.cursor/plans/FE/<module>.plan.md` (или `R2/`) (не наличие файла в src/).
Если статус не completed — остановиться и сообщить пользователю.

## Шаг 3.5 — Прочитать зависимые файлы

Прочитать актуальное содержимое файлов в `src/`, на которые задача ссылается
как на зависимость (компоненты, хуки, api-клиент и т.п.) — ориентироваться на реальный код,
не на описание в их task-файлах.

## Шаг 4 — Реализовать

Создать или изменить только файлы указанные в задаче.
Не трогать другие файлы без явного указания в task-файле.

Инварианты проекта (всегда, без чтения плана) — см. `memory.md`:
- datetime naive MSK — отображать как из API, без конвертации TZ
- секреты не в SPA, `.env` клиента и логи
- ошибки API — `{ error_code, message, details }`; UI показывает `error_code`
- OpenAPI anomaly-api — source of truth для полей REST
- `/usage` — только `GET /agent/usage/runs`; не `usage_total` как primary total

## Шаг 5 — Проверить DoD

Пройти по чеклисту DoD из task-файла.
Если пункт требует теста — запустить тесты ТОЛЬКО по пути(ям), указанным в task-файле
(например `npx vitest run tests/unit/hooks/usePolling.test.ts` или
`npx playwright test tests/e2e/deep-chat.spec.ts`), не полный CI-прогон
`npm run lint && npm run typecheck && npm test && npm run build`.
Если хотя бы один пункт DoD не пройден — ОСТАНОВИТЬСЯ. Не переходить к шагу 6/7.
Сообщить пользователю, что именно не пройдено.

## Шаг 6 — Обновить статус в плане

В YAML-шапке `.cursor/plans/FE/<module>.plan.md` (или `R2/`):
найти запись с нужным id → поменять `status: pending` на `status: completed`.
Менять только поле status, ничего больше в плане не трогать.
Eдинственное место хранения статуса — план модуля.

## Шаг 7 — Закоммитить

Выполнять только если шаг 5 пройден полностью.
Использовать навык `git-commit`.
Сообщение коммита строить по task-id и содержимому:
`feat(область): <что сделано по task-файлу>`
