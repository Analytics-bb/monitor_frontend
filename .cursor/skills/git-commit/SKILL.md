# Протокол коммита

## Шаг 1 — Получить состояние репозитория

```bash
git status
git branch --show-current
git diff --staged
git diff
```

Проверить что текущая ветка соответствует задаче — имя ветки должно совпадать
с типом и slug выполняемой работы. Если ветка `main` или не та — остановиться
и спросить пользователя перед тем как продолжать.

Если ничего не проиндексировано — спросить что именно коммитим,
не добавлять файлы самостоятельно без явного указания.

## Шаг 2 — Сформулировать сообщение

Формат и допустимые типы/области — см. `git.mdc` раздел «Коммиты».

Правила формулировки:
- Описание — что сделано, не как и не зачем
- Повелительное наклонение: «add», «fix», «update» — не «added», «fixed»
- Длина заголовка до 72 символов
- Если diff затрагивает несколько независимых областей — разбить на несколько коммитов

## Шаг 3 — Вывести план коммитов

Показать пользователю до выполнения:
- Список коммитов если их несколько
- Для каждого: сообщение + файлы которые войдут

Пример вывода:
```
коммит 1: feat(tools): add mysql readonly_query tool
  src/tools/mysql.py
  tests/test_mysql.py

коммит 2: docs(tools): document mysql tool usage
  docs/modules/module-3-tools.md
```

## Шаг 4 — Выполнить коммиты

```bash
git add <файлы>
git commit -m "тип(область): описание"
```

Повторить для каждого коммита. После всех вывести `git log --oneline -3`.

## Ограничения

- `git add .` без явного указания — запрещено, только конкретные файлы
- Не изменять существующие коммиты (no amend, no rebase) без явной просьбы
