# Протокол закрытия модуля

Этот skill — gate более высокого уровня, чем task-runner. Он не делает работу
за отдельные задачи, он проверяет, что вся работа модуля закончена и
действительно соответствует DoD плана, а не только сумме DoD отдельных задач.

## Шаг 0 — Проверить чистоту рабочей копии

Если есть незакоммиченные изменения — остановиться, сообщить пользователю.
Закрытие модуля не должно прятать в себе случайный незакоммиченный код.

## Шаг 1 — Прочитать план модуля целиком

Путь: `.cursor/plans/FE/<module-id>.plan.md` (FE) или `.cursor/plans/R2/<module-id>.plan.md` (контракт).
Читать полностью — YAML-шапку (`todos`, `isProject`) и весь текст плана,
особенно секцию `## DoD` (или `## 10. DoD`) и секцию `## 8. Структура`.

В отличие от task-runner, здесь план читается ПОЛНОСТЬЮ всегда — это
финальная проверка, экономия токенов здесь не приоритет.

## Шаг 2 — Проверить, что все todos закрыты

Пройти по списку `todos` в YAML-шапке.

Если хотя бы один `status != completed`:
- ОСТАНОВИТЬСЯ.
- Сообщить пользователю точный список id и content незакрытых задач.
- Не выполнять последующие шаги.

Если список `todos` пуст — это ошибка плана, сообщить пользователю и
остановиться (модуль без задач не может быть закрыт через этот протокол).

## Шаг 3 — Проверить структуру файлов

Сравнить секцию `## Структура` плана (список файлов страницы, компонентов, хуков, тестов)
с фактическим содержимым `src/` и `tests/` (пути из плана: `src/pages/`, `src/components/`,
`src/api/`, `src/hooks/`, `tests/unit/`, `tests/e2e/`).

Зафиксировать как нарушение:
- файлы, заявленные в плане, но отсутствующие в `src/` / `tests/`;
- публичные экспорты или маршруты, не описанные в плане, если план явно ограничивает границы модуля.

Если есть нарушения — ОСТАНОВИТЬСЯ, сообщить список несоответствий,
не выполнять следующие шаги.

## Шаг 4 — Стилевая проверка (ESLint + Prettier + TSDoc)

Источник правил: `.cursor/rules/development.mdc` (ESLint + Prettier — lint + format;
TSDoc на публичных границах, язык — русский).

Определить пути модуля из секции `## Структура` плана (например `src/pages/MonitoringPage.tsx`,
`src/hooks/usePolling.ts`, `tests/unit/...`).

### 4a — ESLint + Prettier + typecheck (блокирует закрытие)

```bash
npm run lint
npm run typecheck
```

Prettier check — по скрипту из `package.json` (например `npm run format -- --check .`
или `npx prettier --check "src/**/*.{ts,tsx}"`).

Предпочитать **scoped** проверку, если план задаёт подмножество путей и команда
поддерживает аргументы (например `npx eslint src/hooks/`).

Если ESLint, Prettier или typecheck вернули ошибки — ОСТАНОВИТЬСЯ, вывести вывод.
Не чинить без явного запроса пользователя.

### 4b — TSDoc (блокирует закрытие; при нарушениях — дописать)

Источник формата: `.cursor/rules/development.mdc` § TSDoc.
Проверить экспортируемые **компоненты, хуки, функции** в путях модуля из плана.

Язык — русский; термины библиотек/протоколов (`React`, `fetch`, `Zod`) — без перевода.

#### Кого проверяем

| Объект | Обязательность |
|--------|----------------|
| Экспортируемый компонент | TSDoc над объявлением |
| Экспортируемый хук | TSDoc |
| Публичная функция API-клиента | TSDoc **полноценный** (см. ниже) |
| Границы модуля | усиленно: `src/api/`, polling-хуки, парсеры ошибок |

Внутренние хелперы и тривиальные обёртки — без TSDoc допустимы,
если поведение очевидно из имени и не на публичной границе.

#### Что считается «полноценным» TSDoc

**Недостаточно** одной строки-ярлыка («Хук polling», «HTTP client») у публичных
функций и хуков на границах модуля.

Минимум для **публичной функции/хука**:

1. **Краткое описание** — что делает (1–2 предложения).
2. **Параметры** — `@param`, если неочевидны из сигнатуры.
3. **Возвращает** — `@returns`, если тип/семантика return неочевидны.
4. **Ошибки** — `@throws` или абзац, если бросает/пробрасывает доменные ошибки.
5. **Побочные эффекты** — отдельным абзацем, если есть сеть, polling, `localStorage`.

#### Действия при нарушениях

1. Составить список: файл, символ, чего не хватает.
2. **Дописать полноценные TSDoc** в коде модуля (не только отчёт).
3. Повторно пройти проверку по списку.
4. Только после этого переходить к Шагу 5.

Если после правок остаются пробелы — ОСТАНОВИТЬСЯ, перечислить файл и символ.
Не закрывать модуль «с долгом» по стилю.

## Шаг 5 — Прогнать полный тест-сьют модуля

Запустить полный набор проверок модуля (не точечные тесты отдельных задач):

```bash
npm run lint && npm run typecheck && npm test && npm run build
```

Если план модуля требует e2e для DoD — дополнительно `npm run test:e2e`
(или scoped: `npx playwright test tests/e2e/<module>.spec.ts`).

Если в проекте есть scoped-команды — ограничить путями из плана
(например `npx vitest run tests/unit/hooks/`).

Если что-то падает — ОСТАНОВИТЬСЯ.
Сообщить, какие проверки красные, не пытаться чинить самостоятельно без
явного запроса пользователя.

## Шаг 6 — Пройти DoD-чеклист плана построчно

Взять секцию `## DoD` плана. Для каждого пункта явно зафиксировать
выполнено / не выполнено, с конкретным основанием (не "выглядит готово").

Механически проверяемые пункты — проверить инструментами:
- grep-правила (например `: any`, `as any` вне разрешённых файлов из плана) — выполнить grep, привести результат;
- секреты в клиентском коде (`CURSOR_API_KEY`, `MCP_ANALYTICS_TOKEN`) — grep по `src/`;
- запрещённые прямые импорты между модулями — сравнить с планом;
- наличие документации (`docs/modules/module-N-*.md`) — проверить существование
  файла, если DoD его требует;
- документация по DoD — **механическая** сверка с `module-doc/template.md`
  (см. ниже); если DoD требует «через module-doc», а файл не проходит —
  **блокировать** закрытие;
- наличие записей в `.env.example` / `package.json`, если DoD их требует.

#### DoD: проверка документации (механическая, блокирует закрытие)

Если DoD содержит «документация через module-doc» / «docs/modules/module-N-*»:

1. Файл `docs/modules/module-<n>-*.md` существует.
2. Все 12 заголовков секций из `module-doc/template.md` присутствуют
   (grep по `^## ` в файле).
3. Подсекции зависимостей: `### Модули проекта` и `### Конфигурация`.
4. Строка «Намеренно не покрыто:» в секции «Тесты» (допустимо `N/A — …`).
5. Табличные заголовки совпадают с шаблоном:
   - `| Аспект | Решение |` в «Публичный интерфейс»
   - `| Сущность | Файл | Описание |` в «Контракты и сущности»
   - `| Код | Константа | Условие | Примечание |` в «Ошибки и коды»
   - `| Файл | Что проверяет | Уровень |` в «Тесты»
6. `last_reviewed` в YAML front matter — актуальная дата.

При нарушении — **не закрывать**; исправить документ через навык `module-doc`
или сообщить пользователю список расхождений.

Семантическая точность (совпадение с кодом/тестами) — ручное подтверждение
на Шаге 7, если агент не перечитывал все тесты модуля в этой сессии.

Если хотя бы один механически проверяемый пункт DoD не пройден —
ОСТАНОВИТЬСЯ, сообщить, что именно не пройдено. Не переходить к Шагу 7.

## Шаг 7 — Подвести итог пользователю и запросить финальное подтверждение

Вывести краткий отчёт:
- все todos: completed (да/нет);
- структура: соответствует (да/нет);
- стиль: ESLint + Prettier + typecheck — зелёные (да/нет); TSDoc — полные по §4b (да/нет, что дописано/пропущено);
- тесты: зелёные (да/нет, какие падали если падали);
- DoD: построчно, выполнено / требует ручного подтверждения;
- memory.md: какая строка будет обновлена (Шаг 8b).

Если ВСЕ автоматические проверки прошли, но есть пункты DoD, требующие
ручного подтверждения (документация, апрув архитектуры и т.п.) — явно
спросить пользователя, подтверждает ли он эти пункты, ПЕРЕД переходом
к Шагу 8. Не закрывать модуль без этого подтверждения.

## Шаг 8 — Закрыть модуль (план + memory + DoD)

Только если все автоматические проверки прошли И пользователь подтвердил
пункты, требующие ручного подтверждения.

Путь к плану: `.cursor/plans/FE/<module-id>.plan.md` или `.cursor/plans/R2/<module-id>.plan.md`.

### 8a — YAML-шапка плана

- сменить `isProject: false` → `isProject: true`
  (или эквивалентный маркер закрытия — сверить с уже закрытыми модулями релиза);
- **не** менять `todos` (они уже `completed`).

### 8b — Секция `## DoD` в теле плана

Для каждого пункта DoD, подтверждённого на Шаге 6, заменить `- [ ]` на `- [x]`.
Не менять текст пунктов, только чекбоксы. Пункты, оставшиеся без подтверждения
на Шаге 7, — не отмечать.

### 8c — `.cursor/rules/memory.md`

Обновить таблицу «Статус страниц и инфраструктуры» для закрытого модуля:

| Было | Стало |
|------|-------|
| `pending`, `in progress` | `done` |
| примечание «Phase N» | убрать или заменить на `docs/modules/...` |

В блоке «Критический путь реализации» — пометить завершённый модуль и выделить следующий.

Не трогать блок «Инварианты» и ADR, если закрытие модуля не меняет архитектуру.
Не дублировать контракты из плана — только статус и навигация.

## Шаг 9 — Закоммитить закрытие

Использовать навык `git-commit`.
В коммит входят: план модуля (8a–8b), `.cursor/rules/memory.md` (8c).

Сообщение коммита:
`chore(<module-id>): close module — all tasks done, DoD passed`

Если в плане модуля были явные ссылки на закрытие пунктов module-0
(например "materializes X contract", "closes M0 §N") — включить их
в тело коммита отдельной строкой, по тексту секции "Артефакты" или
"Зависимости" плана модуля.

## Жёсткие правила (всегда)

- Никогда не закрывать модуль, если хотя бы один todo не completed.
- Никогда не закрывать модуль с ошибками ESLint, Prettier или typecheck в путях модуля.
- Никогда не закрывать модуль с однострочными TSDoc у публичных функций/хуков
  на границах API (см. Шаг 4b); при проверке — дописать полноценные, не только отчёт.
- Никогда не закрывать модуль с красными тестами.
- Никогда не закрывать модуль, если есть незакоммиченные изменения на входе.
- Никогда не пропускать Шаг 7 (подтверждение пользователя) даже если
  все автоматические проверки прошли — закрытие модуля необратимо
  меняет статус, финальное слово должно быть за пользователем.
- Никогда не закрывать модуль без обновления `.cursor/rules/memory.md` (Шаг 8c).
- Никогда не оставлять секцию `## DoD` с неотмеченными пунктами, подтверждёнными
  на Шаге 6 (Шаг 8b).
- Не использовать этот skill для частичного прогресса — это терминальная
  операция для модуля, а не промежуточный чекпоинт.
