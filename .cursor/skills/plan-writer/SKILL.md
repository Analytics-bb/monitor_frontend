# Протокол написания модульного плана

## Шаг 1 — Собрать контекст перед написанием
Прочитать:
- `.cursor/rules/memory.md` — инварианты, текущий релиз и состояние модулей
- Индексный план текущего релиза (M0) — путь из `memory.md`, раздел «Текущий релиз»
- `.cursor/plans/R1/module-0-anomaly-agents.plan.md` — индекс R1 как исторический эталон терминологии
- Соседние планы если модуль граничит с уже написанными (R1: `.cursor/plans/R1/`, R2: `.cursor/plans/R2/`)

Уточнить у пользователя если неясно:
- Что модуль делает и чего явно не делает
- Кто потребитель публичного API
- Какие модули должны быть готовы до этого

## Шаг 2 — Заполнить шаблон
Шаблон: см. `template.md` в этой папке.
Сохранить план в `.cursor/plans/R2/module-N-<slug>.plan.md` (новые планы R2).
Task-пакет — в `.cursor/plans/R2/<module-plan-basename>/` (имя папки = имя
файла плана без `.plan.md`; см. `task-packager`).

Обязательные секции (не пропускать, при неприменимости писать N/A + причину):
- Название, Цель
- Границы: что входит / что НЕ входит
- Ключевые гарантии и инварианты (сверить с memory.md)
- Edge-cases с ожидаемым поведением
- Mermaid-схема
- Человекочитаемый флоу (клиент ↔ сервер, шаги)
- Структура файлов
- Публичный API — с указанием кто потребитель
- Тесты — измеримые, не «проверить что работает»
- DoD — конкретные критерии, не абстрактные
- Зависимости + Артефакты
- YAML-шапка с `todos:` (может быть пустым `[]`)

## Шаг 3 — Проверить согласованность
- Имена сущностей совпадают с M0 §2 — таблица владельцев (путь к M0 из `memory.md`)
- Инварианты из memory.md соблюдены (MySQL read-only, MSK, async, redaction)
- Нет дублирования контрактов из других модулей — только ссылки

## Шаг 4 — Апрув
Спросить: «Остались вопросы по этому модулю?»
Не переходить к следующему без явного подтверждения.

## Уровень детализации
В плане — только «что и почему». Не писать:
- Конкретные значения переменных и сигнатуры функций
- SQL-тексты и шаблоны
- Конкретные входные данные в описании тестов
- Код или псевдокод внутри задач

Детали реализации (паттерн файла, wire в main) — в task-файлах как «Решения /
якоря реализации», не в модульном плане; план даёт контракт и структуру файлов.
