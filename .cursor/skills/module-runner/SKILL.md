# Протокол прогона модуля

## Шаг 0 — Предусловия

- План модуля апрувлен (см. `planning.mdc` — раздел «Апрув — жёсткие ограничения»).
- Task-пакет существует: `.cursor/plans/<release>/<module-plan-basename>/index.md`
  (`<release>` — `FE` или `R2` из `memory.md`; имя папки = имя файла плана без `.plan.md`;
  см. `task-packager`) и
  task-файлы на месте (если нет — сообщить, что нужен task-packager).
- Рабочая копия чистая. Если есть незакоммиченные изменения — остановиться,
  сообщить пользователю.
- Ветка соответствует модулю: `feat/m<N>-<slug>` (см. `development.mdc` —
  раздел «Организация веток»). Если ветка не создана — остановиться,
  сообщить пользователю.

## Шаг 1 — Построить порядок выполнения

Прочитать `index.md` пакета. Построить топологический порядок задач по
`depends_on` (задачи без незакрытых зависимостей — выполнимы сейчас).

Если в графе цикл — остановиться, сообщить пользователю, не пытаться
угадать порядок.

## Шаг 2 — Взять следующую готовую задачу

Готова = все задачи из `depends_on` имеют `todos[*].status: completed`
в YAML-шапке `module-N.plan.md`.

Если готовых задач несколько одновременно и они не помечены в `index.md`
как параллельно-безопасные — выполнять в порядке, заданном планом
(не параллелить самостоятельно без явного указания на параллельность,
см. Шаг 5).

Если готовых задач нет, но есть незавершённые — это означает блокировку
(зависимость не может быть закрыта). Остановиться, сообщить, какая
задача и чем блокирована.

Если задач не осталось — переход к Шагу 4.

## Шаг 3 — Выполнить протокол task-runner для выбранной задачи

Применить полный протокол skill `task-runner` к этой задаче, без
сокращений (включая его Шаг 0 — проверку чистоты рабочей копии, Шаг 5 —
стоп при провале DoD).

- Если task-runner дошёл до коммита успешно — вернуться к Шагу 2,
  взять следующую задачу.
- Если task-runner остановился (DoD не пройден, зависимость не готова,
  грязная рабочая копия) — ОСТАНОВИТЬ весь module-runner. Не пытаться
  перейти к следующей задаче через провалившуюся. Сообщить пользователю
  точную причину остановки и какая задача не выполнена.

Это правило жёсткое: одна провалившаяся задача останавливает весь
автономный цикл модуля, а не только саму задачу.

## Шаг 4 — Все задачи завершены

Когда в плане все todos `status: completed`:
- не закрывать модуль самостоятельно;
- вызвать skill `module-closer`, передав module-id и путь к плану
  (`.cursor/plans/<release>/<module-plan-basename>/` + родительский `module-N.plan.md`).

module-runner не подменяет module-closer — задачный цикл и закрытие
модуля это разные уровни строгости (см. ограничения ниже).

## Шаг 5 — Параллельные ветки (если index.md указывает параллельность)

Если `index.md` явно помечает группу задач как параллельно-безопасную
(нет общих depends_on, нет общего владения одним файлом/контрактом) —
module-runner может сообщить пользователю эту возможность и предложить
вести их в отдельных сессиях/ветках (см. `development.mdc` — раздел
«Параллельная разработка»), но **не запускает параллельное выполнение сам в
рамках одного вызова** — параллельность реализуется отдельными вызовами
агента (отдельные сессии/ветки), не одним потоком module-runner.

## Жёсткие правила

- Никогда не пропускать Шаг 3 целиком ради скорости — каждая задача
  проходит полный протокол task-runner, включая тесты и DoD.
- Никогда не продолжать цикл после остановки task-runner на задаче.
- Никогда не вызывать module-closer, если есть незакрытые todos.
- Не коммитить от своего имени напрямую — коммиты делает только
  task-runner (на уровне задачи) и module-closer (на уровне модуля).
- Между задачами не суммировать/не сокращать инварианты проекта —
  каждый вызов task-runner применяет полный набор инвариантов заново,
  это не оптимизируется в module-runner.
